## Tool Calling: El LLM Decide

En un chat normal, el LLM genera texto. Con **tool calling**, el LLM puede decidir que necesita ejecutar una herramienta para responder. En vez de texto, retorna un `ToolCall` con nombre y argumentos.

<Callout variant="info" title="El LLM no ejecuta nada">
El LLM solo *decide* que herramienta usar y con que argumentos. LocalForge ejecuta la herramienta, obtiene el resultado y se lo devuelve al LLM. Recien entonces el LLM genera la respuesta final para el usuario usando ese resultado real.
</Callout>

## Los Modelos de Datos

Antes de ver el flujo, entendamos las estructuras involucradas. LocalForge define dataclasses simples en `app/skills/models.py`:

```python
@dataclass
class ToolCall:
    name: str                    # Nombre de la herramienta
    arguments: dict[str, Any]    # Argumentos extraidos por el LLM

@dataclass
class ToolResult:
    tool_name: str
    content: str                 # Resultado como string
    success: bool = True         # False si hubo error

@dataclass
class ToolDefinition:
    name: str
    description: str
    parameters: dict[str, Any]   # JSON Schema de los parametros
    handler: Callable[..., Awaitable[str]]  # La funcion async que ejecuta
    skill_name: str | None = None
```

<Concept title="Separacion de concerns">
`ToolCall` es lo que el LLM *pide*. `ToolResult` es lo que LocalForge *devuelve*. `ToolDefinition` es la *registracion* de la herramienta con su handler. El LLM nunca ve los handlers -- solo los schemas JSON.
</Concept>

## ChatResponse: Texto o Tool Calls

El cliente Ollama retorna un `ChatResponse` que puede contener texto, tool calls, o ambos. Definido en `app/llm/client.py`:

```python
@dataclass
class ChatResponse:
    content: str
    tool_calls: list[dict] | None = None
    input_tokens: int | None = None
    output_tokens: int | None = None
    model: str | None = None
    total_duration_ms: float | None = None
```

Dos escenarios posibles:

```python
# Escenario 1: El LLM quiere usar una herramienta
ChatResponse(
    content="",
    tool_calls=[{
        "function": {
            "name": "get_weather",
            "arguments": {"city": "Buenos Aires"}
        }
    }]
)

# Escenario 2: El LLM responde directamente (sin tools)
ChatResponse(content="Hola, en que puedo ayudarte?", tool_calls=None)
```

## chat_with_tools()

Este es el metodo central en `app/llm/client.py` que comunica con la API de Ollama:

```python
async def chat_with_tools(
    self,
    messages: list[ChatMessage],
    tools: list[dict] | None = None,
    model: str | None = None,
    think: bool | None = None,
) -> ChatResponse:
    url = f"{self._base_url}/api/chat"
    use_model = model or self._model

    payload: dict = {
        "model": use_model,
        "messages": self._build_message_dicts(messages),
        "stream": False,
    }

    if tools:
        # think: True es incompatible con tools en qwen3
        payload["tools"] = tools
    elif think is False:
        # Explicit False: skip thinking (clasificacion rapida)
        payload["think"] = False
    elif model is None:
        # Solo habilitar thinking para chat default sin tools
        payload["think"] = True

    resp = await self._http.post(url, json=payload)
    resp.raise_for_status()
    data = resp.json()
    msg = data["message"]
    content = msg.get("content", "")
    tool_calls = msg.get("tool_calls")

    # Strip reasoning blocks: <think>...</think>
    if content:
        content = _RE_THINK.sub("", content)

    return ChatResponse(
        content=content,
        tool_calls=tool_calls,
        input_tokens=data.get("prompt_eval_count"),
        output_tokens=data.get("eval_count"),
        model=use_model,
    )
```

<Callout variant="warning" title="Logica condicional de think">
Nota como `think` se maneja de forma condicional: si hay tools, nunca se envia `think`. Si es `False` explicito, se envia para deshabilitar reasoning. Si no se especifica modelo custom, se habilita thinking por defecto. Esta logica evita la incompatibilidad de qwen3.
</Callout>

## El Parametro think

<Concept title="think: True | False | None">
`think=True`: el modelo razona paso a paso antes de responder (chain-of-thought). `think=False`: respuesta directa sin razonamiento. `think=None`: comportamiento default del modelo. Cuando hay tools en el payload, thinking se omite automaticamente.
</Concept>

| Escenario | think | Por que |
|-----------|-------|---------|
| Chat con tools | *(omitido)* | Incompatibilidad qwen3 + tools + think |
| Guardrails (binary) | `False` | Solo necesita si/no rapido |
| Summarizer | `False` | Respuesta estructurada, no reasoning |
| classify_intent | `False` | Clasificacion rapida sin chain-of-thought |
| Chat libre sin tools | `True` | Razonamiento mejora calidad |
| Problemas complejos | `True` | Chain-of-thought ayuda |

<Callout variant="error" title="Incompatibilidad qwen3 + tools + think">
qwen3 con `think=True` y tools genera outputs inconsistentes -- a veces mezcla reasoning XML con JSON de tool calls. LocalForge resuelve esto automaticamente: cuando `tools` esta presente en el payload, el parametro `think` nunca se incluye.
</Callout>

## Calibracion de Tokens en Tiempo Real

Un detalle sutil: `chat_with_tools()` aprovecha cada respuesta para calibrar la estimacion de tokens:

```python
# Runtime calibration: update chars-per-token ratio
if input_tokens and input_tokens > 0 and not tools:
    try:
        from app.context.token_estimator import calibrate
        prompt_chars = sum(len(d.get("content", "") or "") for d in payload["messages"])
        calibrate(use_model, prompt_chars, input_tokens)
    except Exception:
        pass  # best-effort, never block
```

Esto permite que el estimador de tokens mejore su precision con el uso real, sin agregar latencia.
