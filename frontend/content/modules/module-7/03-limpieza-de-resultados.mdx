## El Problema: Tool Outputs se Acumulan

En cada iteracion del loop, los tool results se agregan al historial de mensajes. Despues de 3-4 iteraciones, el historial puede tener miles de tokens solo en tool outputs.

```
Iteracion 1: web_search -> 2000 chars de resultados
Iteracion 2: get_weather -> 500 chars
Iteracion 3: web_search de nuevo -> 2000 chars mas
Total tool outputs: 4500 chars (~1125 tokens)
```

<Callout variant="warning" title="Presupuesto de contexto">
Cada token gastado en tool outputs viejos es un token menos para historial, memorias y el system prompt. El modelo qwen3.5:9b tiene un contexto de 32K tokens. La limpieza es critica para la calidad de las respuestas.
</Callout>

## _clear_old_tool_results()

La implementacion real en `app/skills/executor.py` usa un patron recomendado por Anthropic -- **tool result clearing**:

```python
def _clear_old_tool_results(
    messages: list[ChatMessage],
    keep_last_n: int = 2,
) -> None:
    """Replace old tool results with compact summaries.

    Keeps the last `keep_last_n` tool messages intact (most recent).
    Once a raw API response is processed, there's no reason to keep
    it verbatim -- the LLM already "vio" esos resultados.
    """
    tool_indices = [i for i, m in enumerate(messages) if m.role == "tool"]

    if len(tool_indices) <= keep_last_n:
        return  # No hay suficientes resultados para limpiar

    for idx in tool_indices[:-keep_last_n]:
        old_content = messages[idx].content
        first_line = old_content.split("\n")[0][:120].strip()
        messages[idx] = ChatMessage(
            role="tool",
            content=f"[Previous result processed -- summary: {first_line}]",
        )
```

<Concept title="Clearing vs Truncation">
LocalForge no simplemente trunca -- extrae la primera linea (hasta 120 chars) como *summary* del resultado viejo. Esto le da al LLM un recordatorio minimo de lo que habia, sin gastar tokens en el contenido completo.
</Concept>

## Cuando se Ejecuta

La limpieza ocurre **despues de cada ronda** de tool calls, antes de la siguiente llamada al LLM:

```python
for iteration in range(MAX_TOOL_ITERATIONS):
    response = await ollama_client.chat_with_tools(working_messages, tools=tools)

    if not response.tool_calls:
        return response.content

    # Ejecutar tools y agregar resultados
    working_messages.append(ChatMessage(role="assistant", ...))
    working_messages.extend(tool_result_map[i] for i in sorted(tool_result_map))

    # Limpiar resultados viejos DESPUES de agregarlos
    _clear_old_tool_results(working_messages, keep_last_n=2)
```

<Callout variant="tip" title="keep_last_n=2">
Se mantienen los ultimos 2 resultados intactos, no solo 1. Esto ayuda cuando el LLM necesita correlacionar el resultado actual con el inmediatamente anterior (por ejemplo, comparar datos de dos busquedas).
</Callout>

## Compactacion JSON (para outputs individuales grandes)

Antes de que un resultado llegue al loop, cada tool output individual pasa por `compact_tool_output` en `app/formatting/compaction.py`. La estrategia tiene 4 niveles:

<Step number={1} title="Verificar tamano">
Si el output es menor al threshold (20,000 chars por defecto), no se toca:

```python
if len(text) <= max_length:
    return text  # Ya es corto, no tocar
```
</Step>

<Step number={2} title="Extraccion JSON (sin LLM)">
Si el output es JSON parseable, se extraen solo los campos clave de forma deterministica:

```python
_JSON_KEY_FIELDS = [
    "name", "full_name", "id", "number", "title",
    "description", "html_url", "url", "state",
    "created_at", "language", "body", "content", "path",
]

def _try_json_extraction(text: str, max_length: int) -> str | None:
    data = json.loads(text)
    # GitHub Search API: {"total_count": N, "items": [...]}
    if isinstance(data, dict) and "items" in data:
        return _extract_item_list(data["items"], max_length)
    # Lista directa: [{"name": "repo1"}, ...]
    if isinstance(data, list) and isinstance(data[0], dict):
        return _extract_item_list(data, max_length)
    # Objeto unico: filtrar campos clave
    if isinstance(data, dict):
        compact = _pick_fields(data)
        return json.dumps(compact, indent=2)
```
</Step>

<Step number={3} title="Compactacion LLM">
Si la extraccion JSON no funciona (texto plano, HTML, etc.), el LLM resume el contenido:

```python
prompt = (
    f"The tool '{tool_name}' returned {len(text)} characters.\n"
    f"The user's request was: \"{user_request}\"\n\n"
    "Summarize, extracting ONLY fields that answer the user's request.\n"
    "CRITICAL RULES:\n"
    "1. PRESERVE EXACT names, IDs, URLs -- NEVER substitute with [placeholder].\n"
    "2. Discard metadata not useful for answering the request.\n"
)
summary = await ollama_client.chat(messages, think=False)
```

<Callout variant="error" title="Riesgo de alucinacion">
El prompt incluye reglas estrictas para que el LLM no reemplace nombres reales con placeholders como `[repo-name]`. Esta regla se agrego despues de detectar que el LLM cambiaba "localforge-assistant" por "[repository]" durante la compactacion.
</Callout>
</Step>

<Step number={4} title="Truncado hard (ultimo recurso)">
Si el LLM falla o retorna vacio, se hace un corte directo:

```python
except Exception as e:
    logger.error("LLM compaction failed: %s", e)
    return text[:max_length] + "\n...[truncated]"
```
</Step>

## Metricas: Configuracion del Threshold

El threshold de compactacion es configurable en `app/config.py`:

```python
class Settings(BaseSettings):
    compaction_threshold: int = 20000  # chars
    max_tools_per_call: int = 8
```

<Callout variant="info" title="20K chars es conservador">
Con un ratio promedio de ~4 chars/token, 20K chars son ~5000 tokens. Esto deja ~27K tokens libres para contexto, historial, system prompt y la respuesta. El valor es tunable via env var `COMPACTION_THRESHOLD`.
</Callout>
