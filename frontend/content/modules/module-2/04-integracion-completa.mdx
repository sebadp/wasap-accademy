## Inicializacion en el Lifespan

Todo comienza en `app/main.py`. El `lifespan` de FastAPI crea y conecta todos los componentes:

```python
# app/main.py (simplificado)
from contextlib import asynccontextmanager
import httpx
from fastapi import FastAPI
from app.config import Settings
from app.llm.client import OllamaClient

@asynccontextmanager
async def lifespan(app: FastAPI):
    settings = Settings()

    # Un solo httpx client para toda la app
    http_client = httpx.AsyncClient(
        timeout=httpx.Timeout(600.0, connect=10.0)
    )

    # Crear el cliente Ollama
    app.state.ollama_client = OllamaClient(
        http_client=http_client,
        base_url=settings.ollama_base_url,  # "http://localhost:11435"
        model=settings.ollama_model,         # "qwen3.5:9b"
    )

    # Crear el cliente WhatsApp
    app.state.whatsapp_client = WhatsAppClient(
        http_client=http_client,
        access_token=settings.whatsapp_access_token,
        phone_number_id=settings.whatsapp_phone_number_id,
    )

    # ... DB, skills, scheduler, memory, etc.

    yield  # La app corre aqui

    # Cleanup
    await http_client.aclose()

app = FastAPI(title="LocalForge", lifespan=lifespan)
```

<Concept title="Lifespan pattern">
FastAPI ejecuta el codigo antes del `yield` al inicio de la app, y el codigo despues del `yield` al apagar. Esto reemplaza los decoradores `@app.on_event("startup")` y `@app.on_event("shutdown")` que estan deprecados.
</Concept>

## El Flujo Completo de un Mensaje

Cuando llega un mensaje de WhatsApp, pasa por varias etapas:

<Step number={1} title="Webhook recibe el payload">
```python
# app/webhook/router.py
@router.post("/webhook")
async def incoming_webhook(
    request: Request,
    background_tasks: BackgroundTasks,
) -> Response:
    body = await request.body()
    signature = request.headers.get("X-Hub-Signature-256", "")

    if not validate_signature(body, signature, settings.whatsapp_app_secret):
        return Response(status_code=200)  # No revelar que fallo

    payload = await request.json()
    messages = extract_messages(payload)
```
</Step>

<Step number={2} title="Validaciones y dedup">
```python
    for msg in messages:
        # Whitelist: solo numeros permitidos
        if msg.from_number not in settings.allowed_phone_numbers:
            continue

        # Rate limiting: maximo N mensajes por ventana
        if not rate_limiter.is_allowed(msg.from_number):
            continue

        # Dedup atomico: INSERT OR IGNORE evita duplicados
        if await repository.try_claim_message(msg.message_id):
            continue  # Ya procesado

        background_tasks.add_task(process_message, msg=msg, ...)
```
</Step>

<Step number={3} title="Procesamiento en background">
El mensaje se procesa como background task para responder 200 inmediatamente a Meta:

```python
async def process_message_generic(
    msg: IncomingMessage,
    platform_client: PlatformClient,
    ollama_client: OllamaClient,
    ...
) -> None:
    # Indicador de typing (emoji reloj en WhatsApp)
    await platform_client.send_typing_indicator(msg.user_id)

    try:
        # ... procesamiento completo
        pass
    finally:
        # SIEMPRE remover el indicador de typing
        await platform_client.remove_typing_indicator(msg.user_id)
```
</Step>

<Callout variant="warning" title="Responder 200 inmediatamente">
Meta reenvia el webhook si no recibis un 200 en 5 segundos. Por eso el procesamiento va en `background_tasks`: el endpoint retorna 200 al instante, y el mensaje se procesa despues.
</Callout>

## Las 4 Fases del Pipeline

LocalForge paraleliza el procesamiento en 4 fases para minimizar latencia:

```python
# Fase A: operaciones iniciales en paralelo
query_embedding, _, daily_logs = await asyncio.gather(
    embed(query),           # Embedding para busqueda semantica
    save_message(msg),      # Guardar mensaje en DB
    load_daily_logs(),      # Cargar logs recientes
)

# Fase B: busquedas en paralelo (dependen de Phase A)
memories, notes, summary, history = await asyncio.gather(
    search_memories(query_embedding),    # Busqueda vectorial
    search_notes(query_embedding),       # Notas relevantes
    get_summary(phone),                  # Resumen de conversacion
    get_recent_messages(phone),          # Ultimos N mensajes
)

# Fase C: clasificacion (ya kickeada como task antes de Phase A)
categories = await classify_task  # ["time", "math", "none", ...]

# Fase D: construir contexto y llamar al LLM
context = _build_context(system_prompt, memories, notes, history, ...)
response = await ollama_client.chat_with_tools(context, tools=selected_tools)
```

<Concept title="Paralelizacion con asyncio.gather">
`asyncio.gather()` ejecuta multiples coroutines en paralelo. En Fase A, el embedding, el guardado en DB, y la carga de logs corren al mismo tiempo. Esto reduce la latencia total de ~3 llamadas secuenciales a ~1 (la mas lenta).
</Concept>

## Dependency Injection via app.state

Todos los componentes se acceden via funciones en `dependencies.py`:

```python
# app/dependencies.py
from fastapi import Request
from app.llm.client import OllamaClient

def get_ollama_client(request: Request) -> OllamaClient:
    return request.app.state.ollama_client

def get_repository(request: Request) -> Repository:
    return request.app.state.repository

def get_settings(request: Request) -> Settings:
    return request.app.state.settings
```

<Callout variant="info" title="Por que no Depends()">
LocalForge usa `get_*(request)` en lugar de `Depends(get_*)` de FastAPI en los handlers porque `process_message` corre como background task, no como endpoint directo. Las dependencias se resuelven en el endpoint y se pasan explicitamente.
</Callout>

## Testear el Sistema Completo

Para tests de integracion, mockeas Ollama y WhatsApp:

```python
# tests/test_integration.py
from unittest.mock import AsyncMock

@pytest.fixture
def mock_ollama():
    client = AsyncMock(spec=OllamaClient)
    client.chat_with_tools.return_value = ChatResponse(
        content="Hola! En que puedo ayudarte?",
        tool_calls=None,
    )
    return client

async def test_full_pipeline(mock_ollama, mock_wa_client):
    """Test end-to-end: mensaje entrante -> LLM -> respuesta."""
    await process_message_generic(
        msg=IncomingMessage(...),
        platform_client=mock_wa_client,
        ollama_client=mock_ollama,
        ...
    )
    mock_wa_client.send_message.assert_called_once()
```

<Callout variant="tip" title="Mockear siempre Ollama en tests">
Nunca llames a Ollama real en tests. Es lento (segundos por llamada), no determinista, y requiere que el modelo este descargado. Usa `AsyncMock(spec=OllamaClient)` para tests rapidos y reproducibles.
</Callout>
