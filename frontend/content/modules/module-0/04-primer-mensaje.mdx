## De punta a punta

Llegó el momento de ver el big picture en acción. Vamos a trazar el camino completo que hace un mensaje de WhatsApp desde que el usuario lo envía hasta que recibe la respuesta de la IA.

<Callout variant="info" title="El corazón de LocalForge">
Este flujo es el corazón del sistema. Todo lo que vas a aprender en los próximos módulos es profundizar en cada uno de estos pasos. Entender el pipeline completo te da el mapa mental para navegar los 85 archivos del proyecto.
</Callout>

## El pipeline completo

<Step number={1} title="Usuario envía mensaje en WhatsApp">
El usuario escribe "Hola, qué puedes hacer?" en el chat de WhatsApp Business. Meta recibe el mensaje en sus servidores.
</Step>

<Step number={2} title="Meta hace POST a tu webhook">
Los servidores de Meta envían un HTTP POST a `https://tu-dominio.ngrok-free.app/webhook` con el payload JSON del mensaje. El POST incluye el header `X-Hub-Signature-256` con la firma HMAC.
</Step>

<Step number={3} title="Verificación de firma + dedup">
LocalForge verifica que el request viene de Meta usando `validate_signature()`. Luego, `try_claim_message()` hace un INSERT OR IGNORE atómico en la tabla `processed_messages` para evitar procesar duplicados (Meta puede reintentar el POST).
</Step>

<Step number={4} title="Parseo del payload + rate limiting">
El parser `extract_messages()` navega el JSON anidado de Meta y extrae el número, texto, message ID y tipo. El `RateLimiter` verifica que el usuario no exceda 10 mensajes por minuto.
</Step>

<Step number={5} title="Typing indicator + background processing">
Se envía una reacción de reloj de arena al mensaje del usuario (typing indicator) y el procesamiento real se despacha a `BackgroundTasks` para retornar 200 inmediatamente a Meta.
</Step>

<Step number={6} title="Fase A — I/O paralelo">
En paralelo: generar embedding del query, guardar el mensaje en SQLite, cargar daily logs. `classify_intent()` se lanza como task para correr en paralelo con el I/O.
</Step>

<Step number={7} title="Fase B — Contexto">
En paralelo: buscar memorias semánticas, buscar notas relevantes, obtener resumen de conversación, cargar historial reciente (últimos 8 mensajes verbatim).
</Step>

<Step number={8} title="Fase C+D — LLM + respuesta">
Se construye el contexto con XML tags, se llama al LLM via Ollama. Si hay tools, se ejecuta el tool calling loop. La respuesta se formatea (Markdown a WhatsApp markup), se splitea si excede 4096 chars, y se envía.
</Step>

## El código del webhook POST

Este es el código **real** de `incoming_webhook` en `app/webhook/router.py`:

```python
# app/webhook/router.py
@router.post("/webhook")
async def incoming_webhook(
    request: Request,
    background_tasks: BackgroundTasks,
) -> Response:
    settings = get_settings(request)
    body = await request.body()

    # 1. Validar firma HMAC-SHA256
    signature = request.headers.get("X-Hub-Signature-256", "")
    if not validate_signature(body, signature, settings.whatsapp_app_secret):
        logger.warning("Invalid webhook signature")
        return Response(status_code=200)  # Siempre 200 para Meta

    payload = await request.json()
    repository = get_repository(request)
    wa_client = get_whatsapp_client(request)

    # 2. Extraer mensajes del payload
    messages = extract_messages(payload)

    for msg in messages:
        # 3. Verificar allowlist
        if msg.from_number not in settings.allowed_phone_numbers:
            logger.warning("Message from non-whitelisted number: %s", msg.from_number)
            continue

        # 4. Rate limiting
        if not rate_limiter.is_allowed(msg.from_number):
            logger.warning("Rate limit exceeded for %s", msg.from_number)
            continue

        # 5. Dedup atómico (INSERT OR IGNORE)
        if await repository.try_claim_message(msg.message_id):
            logger.info("Duplicate message ignored: %s", msg.message_id)
            continue

        # 6. Procesar en background
        background_tasks.add_task(process_message, msg=msg, ...)

    return Response(status_code=200)
```

<Concept title="Siempre retornar 200">
Meta espera un status code 200 para todos los webhooks, incluso si la firma es inválida. Si retornás 403 o 500, Meta reintenta el request hasta 7 veces con backoff exponencial. Por eso LocalForge retorna 200 pero simplemente ignora el payload inválido.
</Concept>

<Concept title="BackgroundTasks">
`BackgroundTasks` de FastAPI permite despachar trabajo asíncrono después de retornar la respuesta HTTP. Esto es crítico: Meta tiene un timeout de 20 segundos para webhooks. Si tu handler tarda más (porque el LLM tarda en responder), Meta marca tu webhook como caído. Con background tasks, retornás 200 en milisegundos y procesás el mensaje en segundo plano.
</Concept>

## El process_message real

Después de retornar 200, `process_message` se ejecuta en background:

```python
# app/webhook/router.py
async def process_message(msg: WhatsAppMessage, ...) -> None:
    """WhatsApp-specific wrapper: convierte a tipos platform-agnostic."""
    incoming = _wa_msg_to_incoming(msg)  # WhatsAppMessage -> IncomingMessage
    adapter = WhatsAppPlatformAdapter(wa_client, msg.message_id)
    await process_message_generic(msg=incoming, platform_client=adapter, ...)


async def process_message_generic(
    msg: IncomingMessage,
    platform_client: PlatformClient,  # Protocol: WhatsApp o Telegram
    ...
) -> None:
    """Platform-agnostic message processor."""
    # Marcar como leído + typing indicator en paralelo
    try:
        await asyncio.gather(
            platform_client.mark_as_read(msg.message_id),
            platform_client.send_typing_indicator(msg.user_id),
        )
    except Exception:
        logger.debug("Failed to send initial platform signals")

    try:
        await _handle_message(msg, settings, platform_client, ...)
    finally:
        # SIEMPRE quitar el typing indicator, incluso si hay error
        try:
            await platform_client.remove_typing_indicator(msg.user_id)
        except Exception:
            pass
```

<Concept title="PlatformClient Protocol">
`PlatformClient` es un Protocol (structural typing de Python). Define métodos como `send_message()`, `mark_as_read()`, `send_typing_indicator()`. Tanto `WhatsAppPlatformAdapter` como `TelegramClient` lo implementan. Esto permite que `process_message_generic` sea 100% reutilizable entre plataformas.
</Concept>

## El dedup atómico explicado

Meta puede enviar el mismo webhook múltiples veces (retries). Sin dedup, procesarías el mismo mensaje dos veces y enviarías respuestas duplicadas:

```python
# app/database/repository.py
async def try_claim_message(self, wa_message_id: str) -> bool:
    """Atomic dedup: returns True if message was ALREADY processed."""
    cursor = await self._conn.execute(
        "INSERT OR IGNORE INTO processed_messages (wa_message_id) VALUES (?)",
        (wa_message_id,),
    )
    await self._conn.commit()
    # Si se insertó (rowcount=1), es nuevo. Si no se insertó (0), ya existía.
    return cursor.rowcount == 0
```

<Callout variant="warning" title="INSERT OR IGNORE vs SELECT + INSERT">
Nunca hagas `SELECT` para verificar y luego `INSERT`. En un sistema async, dos coroutines podrían hacer `SELECT` simultáneamente, ambas ver que no existe, y ambas hacer `INSERT`. `INSERT OR IGNORE` es atómico a nivel de SQLite: solo uno gana.
</Callout>

## El RateLimiter en memoria

```python
# app/webhook/rate_limiter.py
import time
from collections import defaultdict, deque

class RateLimiter:
    def __init__(self, max_requests: int, window_seconds: int):
        self._max_requests = max_requests
        self._window = window_seconds
        self._requests: dict[str, deque[float]] = defaultdict(deque)

    def is_allowed(self, key: str) -> bool:
        now = time.monotonic()
        q = self._requests[key]

        # Limpiar entradas expiradas
        while q and q[0] <= now - self._window:
            q.popleft()

        if len(q) >= self._max_requests:
            return False

        q.append(now)
        return True
```

<Concept title="Sliding Window Rate Limiter">
El rate limiter usa una ventana deslizante con `deque`. Cada request agrega un timestamp. Antes de verificar el límite, limpia los timestamps más viejos que la ventana. `time.monotonic()` se usa en vez de `time.time()` porque es inmune a cambios del reloj del sistema.
</Concept>

<Callout variant="tip" title="Testeabilidad">
Cada una de estas funciones `get_*()` lee de `request.app.state`. En los tests, simplemente asignás mocks a `app.state.repository`, `app.state.ollama_client`, etc. No necesitás montar la infraestructura real.
</Callout>
