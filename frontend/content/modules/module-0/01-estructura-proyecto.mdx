## El mapa del territorio

Antes de escribir una línea de código, necesitás entender el territorio. LocalForge tiene 85+ archivos Python organizados en dominios claramente separados. Cada carpeta es un **bounded context** con una responsabilidad específica.

<Callout variant="info" title="Separación por dominio">
Cada carpeta del proyecto corresponde a un dominio de responsabilidad. Esta separación es intencional y es lo que hace al código mantenible a escala. Un cambio en el sistema de memoria no debería afectar al webhook.
</Callout>

## La estructura real del proyecto

Esta es la estructura **real** de LocalForge, no una versión simplificada:

```
localforge/
  app/
    main.py                 # FastAPI app + lifespan + scheduler jobs
    config.py               # Settings (Pydantic Settings, .env)
    models.py               # Pydantic models compartidos
    dependencies.py         # FastAPI dependency injection
    logging_config.py       # JSON structured logging
    webhook/                # Recepción de mensajes (Módulo 1)
      router.py             # Endpoints GET/POST + process_message
      parser.py             # Extraer mensajes del payload de Meta
      security.py           # Validación de firma HMAC-SHA256
      rate_limiter.py       # Rate limiting por número de teléfono
    llm/                    # Cliente de Ollama (Módulo 2)
      client.py             # OllamaClient (chat + tools + embeddings)
    whatsapp/               # WhatsApp Cloud API
      client.py             # WhatsAppClient (send, mark_as_read, download)
    database/               # Persistencia (Módulo 3)

    conversation/           # Gestión de conversaciones
      manager.py            # ConversationManager + windowed history
      summarizer.py         # Resumen automático de conversaciones largas
    memory/                 # Sistema de memoria (Módulo 4)
```

<Concept title="Bounded Context">
Cada carpeta en `app/` es un bounded context del Domain-Driven Design: tiene su propia responsabilidad, sus propios modelos y su propia lógica. Las dependencias entre contextos fluyen en una sola dirección (el webhook depende del LLM, pero el LLM no sabe nada del webhook).
</Concept>

## Los dominios principales

<Concept title="app/webhook/">
Recibe los mensajes de WhatsApp via HTTP POST. Verifica la firma HMAC-SHA256 de Meta, parsea el payload JSON anidado, aplica rate limiting por número de teléfono y despacha el mensaje al handler correcto via `BackgroundTasks`.
</Concept>

<Concept title="app/llm/client.py">
`OllamaClient` encapsula toda la comunicación con Ollama. Tiene tres métodos principales: `chat()` para conversación, `chat_with_tools()` para tool calling, y `embed()` para generar embeddings. El parámetro `think: bool` controla si el modelo usa chain-of-thought (solo para qwen3, nunca con tools).
</Concept>

<Concept title="app/database/">
SQLite via aiosqlite con sqlite-vec para búsqueda vectorial. `init_db()` retorna una tupla `(conn, vec_available: bool)` para degradar gracefully si sqlite-vec no está disponible. Usa PRAGMAs de performance: `synchronous=NORMAL`, `cache_size=-32000`, `temp_store=MEMORY`.
</Concept>

<Concept title="app/skills/">
Sistema de plugins extensible. Cada skill se define en un archivo SKILL.md con frontmatter parseado con regex (sin PyYAML). El `SkillRegistry` registra tools, y el `executor.py` implementa el tool calling loop con máximo 5 iteraciones.
</Concept>

<Concept title="app/memory/">
Sistema de memoria multinivel: memorias permanentes en SQLite, daily logs en archivos Markdown, session snapshots al hacer `/clear`, y consolidación periódica via LLM para eliminar duplicados.
</Concept>

## El flujo de un mensaje

Cuando un usuario envía un mensaje a LocalForge, pasa por un pipeline parallelizado en 4 fases:

<Step number={1} title="Recepción (webhook/router.py)">
WhatsApp hace un POST a tu endpoint. El handler verifica la firma HMAC-SHA256 con `validate_signature()`, parsea el JSON con `extract_messages()`, aplica rate limiting y dedup atómico con `try_claim_message()`.
</Step>

<Step number={2} title="Fase A — I/O paralelo">
Se ejecutan en paralelo: generar embedding del query, guardar el mensaje en la DB, y cargar daily logs. Además, `classify_intent()` se lanza como `asyncio.create_task` para que corra en paralelo con el I/O.
</Step>

<Step number={3} title="Fase B — Búsqueda de contexto">
Se ejecutan en paralelo: buscar memorias relevantes (semántico), buscar notas relacionadas, obtener resumen de conversación, cargar historial reciente y obtener resumen de proyectos.
</Step>

<Step number={4} title="Fase C+D — Clasificación y generación">
Se construye el contexto completo con `ContextBuilder` (XML tags), se inyecta en el prompt del sistema, y se llama al LLM. Si hay tools activas, el `execute_tool_loop` itera hasta obtener texto o llegar al máximo de iteraciones.
</Step>

```python
# Las 4 fases ejecutan en paralelo dentro de cada grupo
# Ejemplo simplificado de Phase A en router.py:
embed_task = asyncio.create_task(
    ollama_client.embed([user_text], model=settings.embedding_model)
)
save_task = asyncio.create_task(
    repository.save_message(conv_id, "user", user_text, msg.message_id)
)
logs_task = asyncio.create_task(
    daily_log.get_context(days=settings.daily_log_days)
)

# classify_intent corre en paralelo con Phase A
classify_task = asyncio.create_task(
    classify_intent(user_text, ollama_client)
)

# Esperar resultados de Phase A
query_embedding, _, daily_logs = await asyncio.gather(
    embed_task, save_task, logs_task
)
```

## Dependency Injection con FastAPI

LocalForge usa un patrón limpio de dependency injection. Todos los servicios se inicializan en el `lifespan` de FastAPI y se exponen via `app.state`:

```python
# app/dependencies.py — funciones get_*() que FastAPI inyecta
from fastapi import Request
from app.config import Settings
from app.whatsapp.client import WhatsAppClient

def get_settings(request: Request) -> Settings:
    return request.app.state.settings

def get_whatsapp_client(request: Request) -> WhatsAppClient:
    return request.app.state.whatsapp_client

def get_repository(request: Request) -> Repository:
    return request.app.state.repository

def get_ollama_client(request: Request) -> OllamaClient:
    return request.app.state.ollama_client
```

<Concept title="Lifespan Pattern">
FastAPI usa `@asynccontextmanager` para el ciclo de vida de la app. Todo lo que está antes del `yield` se ejecuta al arrancar (inicializar DB, crear clientes HTTP, registrar skills). Todo lo que está después del `yield` se ejecuta al apagar (cerrar conexiones, flush de traces).
</Concept>

## El archivo de modelos compartidos

Los modelos Pydantic que se usan en todo el proyecto están centralizados en `app/models.py`:

```python
# app/models.py
from pydantic import BaseModel

class ChatMessage(BaseModel):
    role: str  # "system", "user", "assistant", or "tool"
    content: str
    images: list[str] | None = None
    tool_calls: list[dict] | None = None

class WhatsAppMessage(BaseModel):
    from_number: str
    message_id: str
    timestamp: str
    text: str
    type: str
    media_id: str | None = None
    reply_to_message_id: str | None = None
```

<Callout variant="tip" title="Convención de nombres">
Usamos `from_number` en vez de `from` porque `from` es una palabra reservada de Python. En el payload de WhatsApp el campo se llama `"from"`, pero en nuestro modelo lo renombramos.
</Callout>
