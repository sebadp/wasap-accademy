## Por qué Docker

LocalForge depende de varios servicios: **Ollama** para los modelos de IA, **ngrok** para exponer tu servidor local a internet, y opcionalmente **Langfuse** (con PostgreSQL, ClickHouse, Redis y MinIO) para observabilidad. Docker Compose los orquesta todos.

<Callout variant="warning" title="Pre-requisito">
Necesitás tener instalado Docker Desktop (Mac/Windows) o Docker Engine + Docker Compose plugin (Linux). Si no lo tenés, instalalo desde https://docs.docker.com/get-docker/ antes de continuar.
</Callout>

## El docker-compose.yml real de LocalForge

A diferencia de los tutoriales que usan PostgreSQL, LocalForge usa **SQLite** como base de datos principal. No necesitás un contenedor de DB. El `docker-compose.yml` tiene tres servicios principales y un stack completo de Langfuse:

```yaml
# docker-compose.yml (servicios principales)
services:
  localforge:
    build: .
    env_file: .env
    volumes:
      - ./data:/app/data          # Persistir SQLite, logs, memorias
      - ./app:/app/app            # Hot-reload del código
      - ./skills:/app/skills      # Skills editables
      - whisper_cache:/home/appuser/.cache/huggingface
    network_mode: host
    restart: unless-stopped
    profiles: ["dev", "prod"]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 5s
      start_period: 60s
      retries: 3

  ollama:
    image: ollama/ollama
    volumes:
      - ollama_data:/root/.ollama
    environment:
      - OLLAMA_HOST=0.0.0.0:11435
    network_mode: host
    restart: unless-stopped
    profiles: ["dev"]

  ngrok:
    image: ngrok/ngrok
    command: http --url=${NGROK_DOMAIN} --log=stdout http://localhost:8000
    environment:
      - NGROK_AUTHTOKEN=${NGROK_AUTHTOKEN}
    network_mode: host
    depends_on:
      - localforge
    restart: unless-stopped
    profiles: ["dev"]
```

<Concept title="network_mode: host">
En vez de crear una red Docker interna, `network_mode: host` hace que los contenedores compartan la red del host. Esto simplifica la comunicación entre Ollama (puerto 11435) y LocalForge (puerto 8000): ambos se ven como `localhost`.
</Concept>

<Concept title="profiles">
Los perfiles de Docker Compose permiten organizar servicios por entorno. `docker compose --profile dev up` levanta los servicios de desarrollo (Ollama, ngrok, Langfuse). En producción, solo levantás el perfil `prod`. Sin especificar perfil, no se levanta ningún servicio.
</Concept>

## El Dockerfile de producción

El Dockerfile de LocalForge sigue buenas prácticas de seguridad:

```dockerfile
FROM python:3.11-slim

RUN apt-get update && apt-get install -y --no-install-recommends \
    # Para faster-whisper (transcripción de audio)
    ffmpeg \
    # Para healthchecks
    curl \
    # Para MCP servers (server-puppeteer, etc.)
    nodejs npm \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

COPY pyproject.toml .
RUN pip install --no-cache-dir --upgrade pip setuptools wheel
RUN pip install --no-cache-dir --no-build-isolation .

# Pre-instalar MCP servers para evitar cold-start timeouts
RUN npm install -g @modelcontextprotocol/server-puppeteer \
    @modelcontextprotocol/server-filesystem \
    @modelcontextprotocol/server-github

COPY app/ app/
COPY skills/ skills/

# Crear usuario no-root (seguridad)
ARG UID=1000
RUN groupadd -g $UID appuser && useradd -u $UID -g $UID -m appuser \
    && mkdir -p /app/data && chown -R appuser:appuser /app
USER appuser

HEALTHCHECK --interval=30s --timeout=5s --start-period=60s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

<Callout variant="info" title="Seguridad: usuario no-root">
El contenedor corre como `appuser` (UID=1000), no como root. Esto es una best practice de seguridad: si alguien compromete la app, no tiene acceso root al contenedor.
</Callout>

## Comandos esenciales

<Step number={1} title="Levantar el entorno de desarrollo">
```bash
# Levantar todos los servicios del perfil dev
docker compose --profile dev up -d

# Verificar que están corriendo
docker compose ps
```
</Step>

<Step number={2} title="Descargar modelos de Ollama">
```bash
# Modelo principal de chat
docker compose exec ollama ollama pull qwen3.5:9b

# Modelo de visión (para analizar imágenes)
docker compose exec ollama ollama pull llava:7b

# Modelo de embeddings (para búsqueda semántica)
docker compose exec ollama ollama pull nomic-embed-text
```
</Step>

<Step number={3} title="Ver logs de la API">
```bash
# Logs en tiempo real (JSON structured)
docker compose logs -f localforge

# Solo errores
docker compose logs -f localforge | grep '"level":"ERROR"'
```
</Step>

<Step number={4} title="Apagar todo">
```bash
# Apagar servicios (los datos persisten en volúmenes)
docker compose --profile dev down

# Reset total: borrar volúmenes (pierdes datos de Ollama y DB)
docker compose --profile dev down -v
```
</Step>

## El stack de Langfuse (observabilidad)

LocalForge incluye un stack completo de **Langfuse v3** para tracing y evaluación. Son 6 servicios adicionales:

```yaml
# Langfuse v3 stack (incluido en docker-compose.yml)
langfuse-web:         # Web UI en http://localhost:3000
langfuse-worker:      # Worker para procesamiento async
langfuse-db:          # PostgreSQL 17 para datos de Langfuse
langfuse-clickhouse:  # ClickHouse para analytics
langfuse-redis:       # Redis para caché y colas
langfuse-minio:       # MinIO (S3-compatible) para blobs
```

<Callout variant="tip" title="Langfuse es opcional">
Langfuse se levanta con el mismo perfil `dev`. Si no necesitás tracing por ahora, podés desactivarlo con `TRACING_ENABLED=false` en tu `.env`. Los servicios de Langfuse no afectan al funcionamiento de LocalForge.
</Callout>

Una vez levantados los servicios:

</Step>

</Step>

<Step number={3} title="Docs de la API">

<Step number={4} title="Langfuse UI (opcional)">

</Step>

## El Makefile de desarrollo

LocalForge incluye un `Makefile` para simplificar los comandos más comunes:

```makefile
VENV := .venv/bin/

dev:        # Crear virtualenv + instalar deps + pre-commit hooks
    python3 -m venv .venv
    $(VENV)pip install -e ".[dev]"
    $(VENV)pre-commit install

test:       # Correr 316+ tests
    $(VENV)pytest tests/ -v

lint:       # Verificar código con ruff
    $(VENV)ruff check app tests

format:     # Formatear código con ruff
    $(VENV)ruff format app tests

typecheck:  # Type checking con mypy
    $(VENV)mypy app

check: lint typecheck test  # Todo junto antes de pushear
```

<Callout variant="tip" title="Siempre antes de pushear">
Corré `make check` antes de cualquier push. Ejecuta lint (ruff) + typecheck (mypy) + tests (pytest) en secuencia. Es lo mismo que ejecuta el CI en GitHub Actions.
</Callout>
