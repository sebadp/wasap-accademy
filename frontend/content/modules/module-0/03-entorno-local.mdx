## El problema de la configuración

Los valores que cambian entre entornos (desarrollo, staging, producción) no van en el código. Van en variables de entorno. LocalForge usa **Pydantic Settings** para gestionarlas con tipado completo y validación automática.

<Callout variant="error" title="Nunca hagas esto">
No hardcodees API keys en el código. No las commitees al repo. Usá `.env` (que está en `.gitignore`) y `.env.example` (que sí se commitea, sin valores reales).
</Callout>

## El .env.example real de LocalForge

El archivo `.env.example` tiene **todas** las variables organizadas por sección. Estas son las más importantes:

```bash
# === WhatsApp Cloud API ===
WHATSAPP_ACCESS_TOKEN=your_access_token_here
WHATSAPP_PHONE_NUMBER_ID=your_phone_number_id_here
WHATSAPP_VERIFY_TOKEN=your_verify_token_here
WHATSAPP_APP_SECRET=your_app_secret_here

# Comma-separated list of phone numbers allowed to interact
ALLOWED_PHONE_NUMBERS=5491112345678

# === Ollama ===
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=qwen3.5:9b

# === Database (SQLite) ===
DATABASE_PATH=data/localforge.db

# === ngrok ===
NGROK_AUTHTOKEN=your_ngrok_authtoken_here
NGROK_DOMAIN=your-domain.ngrok-free.app

# === Logging ===
LOG_LEVEL=INFO
LOG_JSON=true

# === Rate Limiting ===
RATE_LIMIT_MAX=10
RATE_LIMIT_WINDOW=60

# === Audio (Whisper) ===
WHISPER_MODEL=base
WHISPER_DEVICE=cpu

# === Tracing (Langfuse) ===
TRACING_ENABLED=true
LANGFUSE_PUBLIC_KEY=
LANGFUSE_SECRET_KEY=
LANGFUSE_HOST=http://localhost:3000
```

<Callout variant="info" title="ALLOWED_PHONE_NUMBERS">
Este campo es una **allowlist** de números de teléfono. Solo los números listados pueden interactuar con el bot. Es una medida de seguridad básica pero esencial: sin ella, cualquier número de WhatsApp podría consumir tus recursos de LLM.
</Callout>

## Pydantic Settings en acción

El archivo `app/config.py` es el centro neurálgico de la configuración. Mirá cómo se definen las variables con tipos estrictos:

```python
# app/config.py (versión simplificada del real)
from pydantic import field_validator
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    # WhatsApp Cloud API
    whatsapp_access_token: str
    whatsapp_phone_number_id: str
    whatsapp_verify_token: str
    whatsapp_app_secret: str
    allowed_phone_numbers: list[str]

    @field_validator("allowed_phone_numbers", mode="before")
    @classmethod
    def parse_phone_numbers(cls, v: object) -> object:
        if isinstance(v, str):
            return [n.strip() for n in v.split(",") if n.strip()]
        return v

    # Ollama
    ollama_base_url: str = "http://localhost:11434"
    ollama_model: str = "qwen3.5:9b"

    # Database
    database_path: str = "data/localforge.db"
    summary_threshold: int = 40
    history_verbatim_count: int = 8

    # Rate limiting
    rate_limit_max: int = 10
    rate_limit_window: int = 60

    # Audio (Whisper)
    whisper_model: str = "base"
    whisper_device: str = "cpu"
    whisper_compute_type: str = "int8"

    # Vision
    vision_model: str = "llava:7b"

    # Embeddings & Semantic Search
    embedding_model: str = "nomic-embed-text"
    embedding_dimensions: int = 768
    semantic_search_enabled: bool = True
    semantic_search_top_k: int = 10

    # Tracing (Langfuse)
    tracing_enabled: bool = True
    langfuse_public_key: str | None = None
    langfuse_secret_key: str | None = None
    langfuse_host: str = "http://localhost:3000"

    model_config = {"env_file": ".env", "env_ignore_empty": True}
```

<Concept title="field_validator">
El `@field_validator` para `allowed_phone_numbers` convierte un string como `"5491112345678,5491198765432"` en una lista `["5491112345678", "5491198765432"]`. Esto permite que en el `.env` escribas una lista separada por comas (más natural que JSON).
</Concept>

<Concept title="BaseSettings vs BaseModel">
`BaseSettings` hereda de `BaseModel` pero agrega superpoderes: lee automáticamente variables de entorno y archivos `.env`. Si definís `database_path: str`, busca `DATABASE_PATH` (case-insensitive) en el entorno. Los campos con valores por defecto son opcionales; los sin default son **requeridos**.
</Concept>

## Manejo de comentarios en .env

Un detalle de producción interesante: LocalForge tiene un sistema custom para manejar comentarios inline en archivos `.env`:

```python
# app/config.py — manejo robusto de comentarios en .env
def _strip_inline_comment(value):
    """Strip trailing inline shell comments from unquoted env values.
    e.g. 'FIELD=value   # explanation' -> 'value'
    """
    if isinstance(value, str) and "#" in value:
        value = value[: value.index("#")].strip()
    return value

class _SafeEnvSource(EnvSettingsSource):
    def decode_complex_value(self, field_name, field, value):
        return _safe_decode_complex(value)  # strip comment, then JSON decode
```

<Callout variant="warning" title="Por qué esto importa">
Sin este fix, un `.env` con `ALLOWED_PHONE_NUMBERS=123  # mi número` crashea porque Pydantic intenta parsear `"123  # mi número"` como JSON para el tipo `list[str]`. El `_SafeEnvSource` custom hace strip del comentario **antes** de intentar decodificar.
</Callout>

## Validación fail-fast al arrancar

Si falta una variable requerida, Pydantic lanza un `ValidationError` **al arrancar la aplicación**, no cuando la necesitás. Esto es un patrón llamado **fail-fast**:

```
pydantic_settings.errors.SettingsError: 1 validation error
whatsapp_access_token
  Field required [type=missing, ...]
```

<Concept title="Fail-Fast">
El principio fail-fast dice que si algo va a fallar, es mejor que falle lo antes posible. Si tu `.env` tiene un error, preferís saberlo al arrancar (2 segundos después de `docker compose up`) y no 3 horas después cuando un usuario te manda un mensaje y el sistema intenta usar una variable que no existe.
</Concept>

## Cómo se usa Settings en el código

En FastAPI, las settings se inyectan via dependency injection. Nunca se instancian directamente en el código de negocio:

```python
# app/dependencies.py
def get_settings(request: Request) -> Settings:
    return request.app.state.settings

# app/webhook/router.py — uso en un endpoint
@router.post("/webhook")
async def incoming_webhook(request: Request) -> Response:
    settings = get_settings(request)
    # settings.whatsapp_app_secret, settings.allowed_phone_numbers, etc.
```

<Callout variant="tip" title="Una sola instancia">
En el lifespan de FastAPI, `Settings()` se crea **una sola vez** y se guarda en `app.state.settings`. Todas las rutas acceden a la misma instancia via `get_settings()`. Esto evita parsear el `.env` múltiples veces.
</Callout>
