## Qué es el Repository Pattern

El Repository Pattern separa la **lógica de acceso a datos** de la **lógica de negocio**. En vez de escribir SQL en tus handlers de webhook, delegás a un objeto `Repository` que encapsula todas las queries.

<Callout variant="tip" title="¿Por qué importa?">
Sin este patrón, tendrías SQL mezclado con lógica de LLM, formato de mensajes, y envío a WhatsApp. El Repository centraliza todo el acceso a datos en un solo lugar.
</Callout>

## Repository en WasAP

`app/database/repository.py` tiene **más de 100 métodos async**. Aquí los más importantes:

```python
class Repository:
    def __init__(self, conn: aiosqlite.Connection):
        self.conn = conn

    async def save_message(
        self, conversation_id: int, role: str, content: str
    ) -> int:
        cursor = await self.conn.execute(
            "INSERT INTO messages (conversation_id, role, content) "
            "VALUES (?, ?, ?)",
            (conversation_id, role, content),
        )
        await self.conn.commit()
        return cursor.lastrowid

    async def get_active_memories(
        self, phone_number: str, limit: int | None = None
    ) -> list[dict]:
        query = (
            "SELECT id, content, created_at FROM memories "
            "WHERE phone_number = ? AND active = 1 "
            "ORDER BY id DESC"
        )
        if limit:
            query += f" LIMIT {limit}"
        cursor = await self.conn.execute(query, (phone_number,))
        rows = await cursor.fetchall()
        return [dict(row) for row in rows]
```

## Dependency Injection

El Repository se crea una sola vez al inicio y se inyecta via `app.state`:

```python
# En main.py (lifespan)
conn, vec_available = await init_db(settings.database_path)
repository = Repository(conn)
app.state.repository = repository

# En dependencies.py
def get_repository(request: Request) -> Repository:
    return request.app.state.repository
```

<Concept title="app.state">
FastAPI permite guardar objetos en `app.state` durante el lifespan. Esto evita crear una nueva conexión a la DB por cada request — una sola conexión compartida, serializada por aiosqlite.
</Concept>

## Operaciones Atómicas

Un patrón crítico es `get_or_create_conversation()`:

```python
async def get_or_create_conversation(self, phone_number: str) -> int:
    # INSERT OR IGNORE: no falla si ya existe
    await self.conn.execute(
        "INSERT OR IGNORE INTO conversations (phone_number) VALUES (?)",
        (phone_number,),
    )
    # SELECT: siempre retorna el id (nuevo o existente)
    cursor = await self.conn.execute(
        "SELECT id FROM conversations WHERE phone_number = ?",
        (phone_number,),
    )
    row = await cursor.fetchone()
    await self.conn.commit()
    return row["id"]
```

<Callout variant="warning" title="Sin race conditions">
`INSERT OR IGNORE` + `SELECT` es atómico a nivel de SQLite. No necesitás locks ni try/except para manejar inserciones concurrentes.
</Callout>
