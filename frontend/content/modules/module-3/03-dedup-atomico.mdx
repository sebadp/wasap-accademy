## El Problema: Mensajes Duplicados

WhatsApp envia el mismo webhook **multiples veces** si no recibio un HTTP 200 a tiempo. Si procesas cada uno, el usuario recibe la misma respuesta 2 o 3 veces. Este no es un bug -- es un patron estandar en sistemas event-driven.

<Callout variant="warning" title="At-least-once delivery">
WhatsApp Cloud API reintenta webhooks si no recibe acknowledgment en ~5 segundos. Telegram tambien reintenta si no recibe HTTP 200. Cualquier integracion con APIs de mensajeria necesita deduplicacion.
</Callout>

## La Tabla processed_messages

LocalForge usa una tabla dedicada con PRIMARY KEY para garantizar unicidad:

```sql
-- app/database/db.py — SCHEMA
CREATE TABLE IF NOT EXISTS processed_messages (
    wa_message_id TEXT PRIMARY KEY,
    processed_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

<Concept title="PRIMARY KEY como constraint">
El `PRIMARY KEY` en `wa_message_id` crea automaticamente un indice unico. Esto garantiza que dos filas no pueden tener el mismo ID, y la busqueda es O(log n) gracias al B-tree index subyacente.
</Concept>

## La Solucion: INSERT OR IGNORE

El metodo `try_claim_message` combina la verificacion y el marcado en una sola operacion atomica:

```python
# app/database/repository.py
async def try_claim_message(self, wa_message_id: str) -> bool:
    """Atomically claim a message ID. Returns True if already processed (duplicate)."""
    cursor = await self._conn.execute(
        "INSERT OR IGNORE INTO processed_messages (wa_message_id) VALUES (?)",
        (wa_message_id,),
    )
    await self._conn.commit()
    # If rowcount == 0, the INSERT was ignored -> message was already claimed
    return cursor.rowcount == 0
```

<Callout variant="info" title="Retorno invertido">
Nota que `try_claim_message` retorna `True` si el mensaje **ya fue procesado** (es duplicado). Cuando `rowcount == 0`, significa que el INSERT fue ignorado porque el `wa_message_id` ya existia.
</Callout>

## Dedup Adicional: wa_message_id en messages

Ademas de `processed_messages`, LocalForge tiene una segunda linea de defensa. La tabla `messages` tiene un UNIQUE constraint en `wa_message_id`:

```sql
CREATE TABLE IF NOT EXISTS messages (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id INTEGER NOT NULL REFERENCES conversations(id),
    role            TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    content         TEXT NOT NULL,
    wa_message_id   TEXT UNIQUE,  -- Segunda linea de dedup
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Y el Repository tiene un metodo adicional:

```python
async def is_duplicate(self, wa_message_id: str) -> bool:
    cursor = await self._conn.execute(
        "SELECT 1 FROM messages WHERE wa_message_id = ?",
        (wa_message_id,),
    )
    return await cursor.fetchone() is not None
```

## Uso en el Webhook Handler

Asi se usa en el flujo principal de procesamiento:

```python
# app/webhook/router.py — process_message()
async def process_message(message: WhatsAppMessage, ...):
    # Primer paso: dedup atomico
    if await repository.try_claim_message(message.message_id):
        logger.info("Duplicate message ignored", message_id=message.message_id)
        return

    # Si llegamos aca, es la primera vez que vemos este mensaje
    await _handle_message(message, ...)
```

## Por que NO usar SELECT + INSERT

Una alternativa comun pero **incorrecta** seria:

```python
# MAL: Race condition entre SELECT y INSERT
exists = await repository.check_if_exists(message_id)
if exists:
    return
await repository.mark_as_processed(message_id)  # Otro request puede llegar aqui!
```

<Callout variant="error" title="Race condition">
Entre el SELECT y el INSERT, otro request puede llegar con el mismo `message_id`. Ambos pasan el check y procesan el mensaje dos veces. `INSERT OR IGNORE` elimina esta ventana porque la verificacion y la insercion son una sola operacion atomica a nivel del motor SQLite.
</Callout>

## El Patron Completo: 4 Pasos

<Step number={1} title="Recibir webhook">
WhatsApp/Telegram envia el payload al endpoint POST /webhook.
</Step>

<Step number={2} title="Responder 200 inmediatamente">
Antes de procesar, respondemos HTTP 200 para evitar que la plataforma reintente. El procesamiento real se hace en background.
</Step>

<Step number={3} title="INSERT OR IGNORE (try_claim_message)">
Intentamos insertar el `wa_message_id`. Si `rowcount == 0`, el INSERT fue ignorado y paramos.
</Step>

<Step number={4} title="Procesar si es nuevo">
Solo si el INSERT fue exitoso (`rowcount == 1`), procesamos el mensaje completo: transcripcion, LLM, respuesta.
</Step>

<Concept title="Idempotencia end-to-end">
Este patron hace que todo el pipeline sea **idempotente**: no importa cuantas veces llegue el mismo webhook, el resultado es siempre el mismo. El usuario recibe exactamente una respuesta. Este es un principio fundamental de sistemas distribuidos.
</Concept>

## Limpieza de processed_messages

Con el tiempo, la tabla `processed_messages` crece. LocalForge no implementa limpieza automatica porque el tamano es minimo (un TEXT ID + un timestamp por mensaje). Para apps de alto volumen, podrias agregar:

```sql
-- Limpiar mensajes procesados hace mas de 7 dias
DELETE FROM processed_messages
WHERE processed_at < datetime('now', '-7 days');
```
