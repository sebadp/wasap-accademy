## El Pipeline de Indexación

Cuando se crea una memoria o nota, LocalForge la convierte en un embedding y la almacena en sqlite-vec para búsqueda posterior:

```
Texto → ollama_client.embed([text]) → [768 floats] → struct.pack → blob → sqlite-vec
```

<Callout variant="info" title="Indexación automática">
No necesitás indexar manualmente. LocalForge indexa automáticamente cada `/remember`, nota nueva, fact extraído por flush, y nota de proyecto. Todo pasa por el mismo pipeline.
</Callout>

## sqlite-vec: Tablas Virtuales

sqlite-vec es una extensión de SQLite que agrega búsqueda vectorial. Se carga al iniciar la app en `init_db()`:

```python
# app/database/db.py — Carga de sqlite-vec
async def init_db(db_path: str, embedding_dims: int = 768) -> tuple[aiosqlite.Connection, bool]:
    conn = await aiosqlite.connect(db_path, check_same_thread=False)

    # ... schema setup ...

    # Try to load sqlite-vec
    vec_available = False
    try:
        import sqlite_vec

        ext_path = sqlite_vec.loadable_path()
        conn._connection.enable_load_extension(True)
        conn._connection.load_extension(ext_path)
        conn._connection.enable_load_extension(False)

        # Create vector tables
        await conn.execute(VEC_SCHEMA_MEMORIES.format(dims=embedding_dims))
        await conn.execute(VEC_SCHEMA_NOTES.format(dims=embedding_dims))
        await conn.execute(VEC_SCHEMA_PROJECT_NOTES.format(dims=embedding_dims))
        await conn.execute(VEC_SCHEMA_TOOLS.format(dims=embedding_dims))
        vec_available = True
    except Exception:
        logger.warning("sqlite-vec not available, semantic search disabled")

    return conn, vec_available  # El flag controla el path de búsqueda
```

Las tablas virtuales se definen con la sintaxis `vec0`:

```sql
-- Tabla virtual para embeddings de memorias
CREATE VIRTUAL TABLE IF NOT EXISTS vec_memories
  USING vec0(memory_id INTEGER PRIMARY KEY, embedding float[768]);

-- Tabla virtual para embeddings de notas
CREATE VIRTUAL TABLE IF NOT EXISTS vec_notes
  USING vec0(note_id INTEGER PRIMARY KEY, embedding float[768]);

-- Tabla virtual para embeddings de notas de proyecto
CREATE VIRTUAL TABLE IF NOT EXISTS vec_project_notes
  USING vec0(note_id INTEGER PRIMARY KEY, embedding float[768]);

-- Tabla virtual para embeddings de descriptions de tools
CREATE VIRTUAL TABLE IF NOT EXISTS vec_tools
  USING vec0(tool_name TEXT PRIMARY KEY, embedding float[768]);
```

<Concept title="vec_available: flag de degradación grácil">
`init_db()` retorna `(connection, vec_available: bool)`. Si sqlite-vec no se puede cargar (binario faltante, arquitectura incompatible), `vec_available = False` y la app usa búsqueda por recencia en vez de semántica. La app **nunca crashea** por falta de sqlite-vec.
</Concept>

## Serialización: struct.pack

sqlite-vec espera los vectores como **blobs binarios**. Se serializan con `struct.pack`:

```python
# app/database/repository.py
import struct

class Repository:
    @staticmethod
    def _serialize_vector(vec: list[float]) -> bytes:
        return struct.pack(f"{len(vec)}f", *vec)
        # 768 floats × 4 bytes = 3072 bytes por embedding
```

<Callout variant="tip" title="¿Por qué struct.pack y no JSON?">
Un embedding como JSON (`[0.12, -0.45, ...]`) ocuparía ~6KB de texto. El blob binario ocupa exactamente 3072 bytes (768 floats x 4 bytes). Es 2x más compacto y sqlite-vec lo procesa directamente sin parseo.
</Callout>

## embed_memory(): Indexar una Memoria

```python
# app/embeddings/indexer.py
_MAX_EMBED_CHARS = 6_000  # Límite conservador para nomic-embed-text

async def embed_memory(
    memory_id: int,
    content: str,
    repository: Repository,
    ollama_client: OllamaClient,
    model: str,
) -> None:
    """Compute and store embedding for a single memory. Best-effort."""
    try:
        embeddings = await ollama_client.embed(
            [content[:_MAX_EMBED_CHARS]], model=model
        )
        await repository.save_embedding(memory_id, embeddings[0])
    except Exception:
        logger.warning("Failed to embed memory %d", memory_id, exc_info=True)
        # Best-effort: la memoria existe sin embedding, no se propaga el error
```

El `save_embedding` en el repository hace un DELETE + INSERT (sqlite-vec no soporta UPSERT):

```python
# app/database/repository.py
async def save_embedding(
    self, memory_id: int, embedding: list[float], *, auto_commit: bool = True
) -> None:
    blob = self._serialize_vector(embedding)
    # vec0 virtual tables don't support INSERT OR REPLACE — delete first
    await self._conn.execute(
        "DELETE FROM vec_memories WHERE memory_id = ?", (memory_id,)
    )
    await self._conn.execute(
        "INSERT INTO vec_memories (memory_id, embedding) VALUES (?, ?)",
        (memory_id, blob),
    )
    if auto_commit:
        await self._conn.commit()
```

## Triggers de Indexación

<Step number={1} title="/remember texto">
El comando guarda la memoria en SQLite y llama a `embed_memory()` para indexar el embedding.
</Step>

<Step number={2} title="/forget query">
El comando desactiva la memoria y llama a `repository.delete_embedding()` para limpiar el vector.
</Step>

<Step number={3} title="Flush pre-compactación">
Antes de borrar mensajes viejos, el summarizer extrae facts y los indexa con `embed_memory()`.
</Step>

<Step number={4} title="Consolidación">
El consolidator borra embeddings de memorias duplicadas que se mergearon.
</Step>

<Step number={5} title="Startup (backfill)">
Al arrancar, `backfill_embeddings()` indexa memorias que no tienen embedding.
</Step>

<Callout variant="warning" title="embed_note y embed_project_note">
El indexer tiene funciones análogas para notas (`embed_note`) y notas de proyecto (`embed_project_note`). El pipeline es idéntico: texto truncado a 6000 chars, embed via Ollama, guardar blob en la tabla vec correspondiente.
</Callout>
