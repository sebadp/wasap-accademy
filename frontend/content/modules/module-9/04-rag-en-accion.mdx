## RAG: Retrieval-Augmented Generation

RAG es el patrón de **buscar información relevante** antes de que el LLM genere una respuesta. En vez de depender solo del conocimiento interno del modelo (que puede estar desactualizado o no conocer datos del usuario), le inyectamos contexto real extraído de la base de datos.

<Concept title="RAG en una oración">
**R**etrieval (buscar datos relevantes) **A**ugmented (enriquecer el prompt) **G**eneration (generar respuesta con el LLM). El modelo produce mejores respuestas porque tiene información personalizada y actualizada en su contexto.
</Concept>

## El Pipeline Completo en 4 Fases

<Step number={1} title="Embed: convertir el query en vector">
El mensaje del usuario se convierte en un vector de 768 dimensiones via Ollama. Este vector se usará para todas las búsquedas semánticas.
</Step>

<Step number={2} title="Search: buscar en paralelo">
Con ese vector se buscan memorias similares, notas similares, y notas de proyecto relevantes. Todo en paralelo con `asyncio.gather`.
</Step>

<Step number={3} title="Inject: construir el contexto">
Los resultados de búsqueda se formatean y se inyectan en el system prompt dentro de tags XML usando `ContextBuilder`.
</Step>

<Step number={4} title="Generate: el LLM responde">
El LLM genera la respuesta usando todo el contexto inyectado. Conoce las memorias del usuario, sus notas, y el historial de conversación.
</Step>

<Callout variant="info" title="ConversationContext centraliza todo">
En vez de hacer 8 queries separadas en el router, `ConversationContext.build()` encapsula toda la recolección de datos. Internamente ejecuta las búsquedas en paralelo y retorna un objeto con todos los campos listos.
</Callout>

## El Contexto Ensamblado: _build_context()

Los datos de `ConversationContext` se formatean en un system prompt con XML tags:

```python
# app/webhook/router.py
def _build_context(
    system_prompt, memories, relevant_notes, daily_logs,
    skills_summary, summary, history, projects_summary, project_notes,
) -> list[ChatMessage]:
    builder = ContextBuilder(system_prompt)
    builder.add_section("user_memories", _format_memories(memories))
    builder.add_section("active_projects", projects_summary)
    builder.add_section("project_notes", _format_project_notes(project_notes or []))
    builder.add_section("relevant_notes", _format_notes(relevant_notes))
    builder.add_section("recent_activity", daily_logs)
    builder.add_section("capabilities", skills_summary)
    if summary:
        builder.add_section("conversation_summary",
                            f"Previous conversation summary:\n{summary}")

    context = [ChatMessage(role="system", content=builder.build_system_message())]
    context.extend(history)
    return context
```

## Sin RAG vs Con RAG

```
Pregunta del usuario: "¿Cómo iba lo del deploy?"

─── Sin RAG ───
LLM: "No tengo información sobre un deploy. ¿Podés darme más contexto?"

─── Con RAG ───
memories → "Trabaja en fintech, prefiere respuestas técnicas"
notes → "[42] Deploy Guide: Docker Compose con nginx"
projects → "API v2: 3/5 tasks (60%)"
summary → "Discutimos autenticación JWT para la API"

LLM: "Según tu nota de deploy, la estrategia es Docker Compose con nginx como
      reverse proxy. Para la API v2 que venimos armando (ya tenés 3 de 5 tareas
      listas), recomendaría configurar health checks en el compose..."
```

<Callout variant="tip" title="Personalización automática">
Con RAG, el asistente sabe quién sos, en qué trabajás, qué discutiste antes, y qué herramientas tiene disponibles. Sin que repitas contexto cada vez. La calidad de respuesta es dramáticamente mejor.
</Callout>

## Formateo de Datos para Inyección

Cada tipo de dato tiene su función de formateo:

```python
# app/webhook/router.py
def _format_memories(memories: list[str]) -> str | None:
    if not memories:
        return None
    return "Important user information:\n" + "\n".join(f"- {m}" for m in memories)

def _format_notes(relevant_notes: list[Note]) -> str | None:
    if not relevant_notes:
        return None
    return "Relevant notes:\n" + "\n".join(
        f"- [{n.id}] {n.title}: {n.content[:200]}" for n in relevant_notes
    )

def _format_project_notes(project_notes: list) -> str | None:
    if not project_notes:
        return None
    lines = ["Project documents and notes:"]
    for n in project_notes:
        header = f"[{n.id}]"
        if n.title:
            header += f" {n.title}"
        lines.append(f"- {header}: {n.content[:500]}")
    return "\n".join(lines)
```

<Callout variant="warning" title="Truncamiento de notas">
Las notas regulares se truncan a 200 caracteres (`n.content[:200]`) y las notas de proyecto a 500 caracteres (`n.content[:500]`). Esto es intencional: evita que una nota muy larga consuma todo el presupuesto de tokens del contexto.
</Callout>
